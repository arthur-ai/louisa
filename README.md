Here's a summary of how Louisa works end-to-end:

Trigger: A GitHub webhook fires when a release is published on the Arthur Evals Engine repo. Only non-draft, non-prerelease events are processed.

Pipeline:

Signature verification — the incoming webhook payload is validated against your GITHUB_WEBHOOK_SECRET using HMAC-SHA256 with timing-safe comparison.
Diff resolution — Louisa finds the previous published release tag and uses the GitHub Compare API to get all commits between the two tags. If it's the first release, it grabs the last 50 commits.
PR enrichment — For each commit SHA, Louisa fetches associated merged pull requests, deduplicates them, and extracts titles, descriptions, labels, and authors.
Claude analysis — All the raw data is sent to Claude Sonnet with a carefully crafted system prompt. The prompt instructs Claude to categorize changes (features, fixes, breaking changes, etc.), translate code-level details into user-facing impact statements, and output clean GitHub-flavored Markdown.
Release update — The generated notes are written back to the GitHub release via the Releases API, with a small "generated by Louisa" footer.

Mar 9
Here's a complete summary of Louisa as it stands right now.

What Louisa does

Louisa is an automated release notes bot for the Arthur Evals Engine. Every time your team publishes a release on the Evals Engine GitHub repo, Louisa intercepts the event, gathers all the changes since the last release, sends them to Claude for analysis, and writes polished, user-facing release notes back onto the GitHub release page — all within a few seconds, with no manual work required.

How the pieces connect

The system has three services talking to each other: GitHub, a Vercel serverless function, and the Anthropic API. GitHub sends a webhook event to Vercel whenever a release is published. The Vercel function (Louisa) then makes several calls back to the GitHub API to gather commit and PR data, sends that data to Claude for summarization, and finally writes the generated notes back to the release via the GitHub API.

Where everything lives

The code lives in the arthur-ai/louisa repo on GitHub. It's deployed to Vercel under Ashley's projects at https://louisa-three.vercel.app. The serverless function entry point is at /api/webhook, so the full webhook URL is https://louisa-three.vercel.app/api/webhook. There's no database, no frontend, and no dependencies beyond Node built-ins — everything runs through direct fetch calls.

The codebase

There are four source files. api/webhook.js is the main handler — it receives the webhook, validates it, orchestrates the pipeline, and returns a response. lib/crypto.js handles webhook signature verification using HMAC-SHA256 to ensure requests are actually from GitHub. lib/github.js contains all the GitHub API interactions: comparing tags, fetching commits, looking up merged PRs, finding the previous release, and updating the release body. lib/claude.js builds the prompt and calls the Anthropic API, asking Claude to transform raw commit and PR data into organized, user-facing release notes.

The pipeline step by step

When a release is published on the Evals Engine repo, GitHub sends a POST request to the webhook URL. Louisa first verifies the signature against the shared secret to confirm the request is legitimate. It then checks that the event is a published, non-draft, non-prerelease release — anything else gets a 200 with a "skipped" message.

For valid releases, Louisa calls the GitHub API to find the previous published release tag. It then uses the Compare API to get all commits between the two tags. For each commit SHA, it looks up associated merged pull requests, deduplicating them and extracting titles, descriptions, labels, and authors.

All of this raw data gets sent to Claude with a system prompt that instructs it to categorize changes into sections like New Features, Improvements, Bug Fixes, and Breaking Changes. The prompt tells Claude to focus on user impact rather than code-level details, to omit internal noise like CI changes and merge commits, and to output clean GitHub-flavored Markdown with PR links.

Claude's response gets wrapped with a header and a "Release notes generated by Louisa" footer, then Louisa writes it back to the release using the GitHub Releases API PATCH endpoint.

Environment variables

Five variables are configured in both .env.local (for local development) and the Vercel dashboard (for production). GITHUB_TOKEN is a fine-grained personal access token with read/write access to the Evals Engine repo. GITHUB_WEBHOOK_SECRET is a shared secret generated with openssl rand -hex 32 that both GitHub and Louisa use to verify webhook authenticity. ANTHROPIC_API_KEY is the API key for calling Claude. GITHUB_REPO_OWNER is set to arthur-ai and GITHUB_REPO_NAME is set to the Evals Engine repo name.

The webhook

A webhook is configured on the Evals Engine repo (not the louisa repo) under Settings → Webhooks. It points to https://louisa-three.vercel.app/api/webhook, uses application/json content type, and is configured to fire only on Release events. The secret matches the GITHUB_WEBHOOK_SECRET environment variable.

How to redeploy after changes

Push code changes to the arthur-ai/louisa repo, then run vercel --prod from the louisa/ directory locally. The GitHub integration isn't fully connected due to org permissions, so auto-deploys on push aren't active — manual deploys via the CLI are the current workflow.

    const res = await fetch(url, { headers: headers() });
    if (!res.ok) {
      console.error("Compare API error:", res.status, await res.text());
      return [];
    }
    const data = await res.json();
    return (data.commits || []).map(formatCommit);
  }

  const url = `${GITHUB_API}/repos/${owner}/${repo}/commits?sha=${toTag}&per_page=50`;
  const res = await fetch(url, { headers: headers() });
  if (!res.ok) return [];
  const data = await res.json();
  return data.map(formatCommit);
}

export async function getPullRequestsForCommits(owner, repo, commitShas) {
  const prs = new Map();

  const batches = [];
  for (let i = 0; i < commitShas.length; i += 10) {
    batches.push(commitShas.slice(i, i + 10));
  }

  for (const batch of batches) {
    const results = await Promise.all(
      batch.map(async (sha) => {
        const url = `${GITHUB_API}/repos/${owner}/${repo}/commits/${sha}/pulls`;
        const res = await fetch(url, { headers: headers() });
        if (!res.ok) return [];
        return res.json();
      })
    );

    for (const prList of results) {
      for (const pr of prList) {
        if (pr.merged_at && !prs.has(pr.number)) {
          prs.set(pr.number, {
            number: pr.number,
            title: pr.title,
            body: (pr.body || "").slice(0, 1000),
            author: pr.user?.login,
            labels: (pr.labels || []).map((l) => l.name),
            url: pr.html_url,
          });
        }
      }
    }
  }

  return [...prs.values()];
}

export async function getPreviousReleaseTag(owner, repo, currentTag) {
  const url = `${GITHUB_API}/repos/${owner}/${repo}/releases?per_page=10`;
  const res = await fetch(url, { headers: headers() });
  if (!res.ok) return null;

  const releases = await res.json();
  const published = releases
    .filter((r) => !r.draft)
    .sort((a, b) => new Date(b.published_at) - new Date(a.published_at));

  const idx = published.findIndex((r) => r.tag_name === currentTag);
  if (idx >= 0 && idx < published.length - 1) {
    return published[idx + 1].tag_name;
  }

  if (idx === -1 && published.length > 0) {
    return published[0].tag_name;
  }

  return null;
}

}

export async function createRelease(owner, repo, tag, name, body) {
  const url = `${GITHUB_API}/repos/${owner}/${repo}/releases`;
  console.log(`Louisa: creating release for tag ${tag}`);

    method: "POST",
    headers: { ...headers(), "Content-Type": "application/json" },
    body: JSON.stringify({
      tag_name: tag,
      name: name,
      body: body,
      draft: false,
      prerelease: false,
    }),
  });

  if (!res.ok) {
    const err = await res.text();
    throw new Error(`Failed to create release for ${tag}: ${res.status} — ${err}`);

}

}

# Louisa 🤖

**Automatic, AI-powered release notes for GitHub — so your team never has to write them again.**

Louisa is a lightweight serverless bot that listens for new tags and releases on your GitHub repo, analyzes the commits and pull requests since the last release, and uses Claude to generate polished, user-facing release notes — published directly to your GitHub Releases page.

No manual steps. No copy-pasting changelogs. Just push a tag and Louisa handles the rest.

---

## How It Works

```
Tag pushed to GitHub
        │
        ▼
GitHub webhook fires ──► Vercel serverless function
        │
        ├─► Verifies webhook signature (HMAC-SHA256)
        ├─► Fetches commits between this tag and the previous release
        ├─► Fetches merged pull requests for those commits
        ├─► Sends everything to Claude for analysis and summarization
        └─► Creates a published GitHub Release with formatted notes
```

Louisa handles two scenarios:

1. **Tag push** — When a new tag is pushed, Louisa automatically creates a published release with generated notes. No one needs to touch GitHub Releases manually.
2. **Manual release** — If someone creates a release by hand, Louisa detects it and fills in the release notes if they're empty.

---

## What You Get

Louisa generates release notes that are:

- **Grouped by product area** — not by change type. Sections like "Evaluation & Experiment Enhancements" instead of "Bug Fixes" and "Features."
- **Written for users** — every bullet leads with the benefit or capability, not the code change.
- **Clean and consistent** — follows a structured format with section summaries, bold feature names, and horizontal dividers.
- **Free of internal noise** — CI changes, merge commits, and refactors are filtered out automatically.

## Architecture

Louisa is intentionally simple — no frameworks, no SDKs, no database. Everything runs in a single Vercel serverless function using native `fetch` calls to the GitHub and Anthropic APIs.

| Component | Purpose |
|-----------|---------|
| `api/webhook.js` | Receives GitHub webhooks, routes between tag and release events, orchestrates the pipeline |
| `lib/crypto.js` | Verifies webhook authenticity using HMAC-SHA256 with timing-safe comparison |
| `lib/github.js` | Compares tags, fetches commits, resolves merged PRs, creates and updates releases |
| `lib/claude.js` | Builds the prompt and calls Claude to transform raw data into formatted release notes |

## Troubleshooting

**Webhook returns 401 (Invalid signature)**
The `GITHUB_WEBHOOK_SECRET` in Vercel doesn't match the secret configured on the GitHub webhook. Make sure they're identical.

**Release notes aren't appearing**
Check the Vercel function logs at **vercel.com → Project → Deployments → Latest → Functions**. Common causes:
- GitHub token doesn't have write access to the target repo
- Environment variables aren't set in Vercel (or weren't redeployed after adding them)
- The webhook isn't firing the right events (check Recent Deliveries on the webhook page)

**Notes appear but are empty or generic**
This usually means 0 commits were found between tags. Check that the previous release tag exists and that commits were made between the two tags.

**Duplicate releases**
Louisa checks for existing releases before creating one and skips if notes are already present. If you see duplicates, ensure the webhook isn't configured on multiple repos or that multiple webhook entries don't exist for the same URL.
